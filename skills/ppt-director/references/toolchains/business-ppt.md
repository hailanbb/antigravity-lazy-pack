# Toolchain: business-ppt

## Use When

业务汇报、数据分析、政务汇报、国企项目汇报、培训课件、项目总结。

## Priority

1. 逻辑清晰。
2. 数据准确。
3. 标题有观点。
4. 版式稳定。
5. 风格一致。
6. 设计语言对表清楚。
7. 最终布局审查通过。

## Recommended Execution

- 优先生成标准交付文档和页面结构导演稿。
- 在 HTML 生成前必须先执行页面视觉导演优化，把内容型 MD 升级为可设计的页面结构稿。
- 页面结构导演稿只描述内容结构、模块排布和表达重点，不写具体设计语言。
- 页面结构导演稿里的区域词只用于排版，生成时必须清洗 `左侧：`、`右侧：`、`顶部：`、`底部：`、`中部：`、`侧边：` 等前缀，不能把它们作为模块标题上屏。
- 生成前必须确认设计语言，并读取 `design.md`、style-card、slide-type-map 和 contact sheet。
- 生成 HTML/PPTX 前必须合成“生成就绪导演稿”，明确输出格式、画布参数、区域坐标/比例、设计系统 token、组件映射、连接规则和生成优先级。
- 不要求所有页面与截图版式一致，但必须符合设计文档的视觉 DNA；能靠齐样例页型时尽量靠齐。
- 默认先生成 HTML 设计预览、每页截图和 contact sheet。
- HTML 预览必须按 `references/workflows/final-layout-review.md` 审查：产品解决方案专家看业务价值、场景闭环、方案边界和可信抓手；PPT 专家看视觉重心、饱满度、结构、密度和主视觉表达。
- HTML 审查总分低于 80 或存在 P0 问题时，不进入 PPTX 映射。
- HTML 审查通过并经用户确认后，再使用 python-pptx、artifact-tool 或 Marp 生成/映射文件。
- PPTX 生成后必须导出实际渲染截图，与 HTML/PNG 预览做视觉一致性对比，并再次执行最终布局审查，通过后才交付。
- 数据图表使用原始数据生成，不手工编造。
- 图片没有准备好时使用明确占位符路径。
- 每个页面函数化，方便局部修改。
- 代码中优先抽象母版函数和组件函数，避免每页自由发挥成不同风格。

## Coding Agent Prompt Addendum

```text
这是业务型 PPT。请优先保证逻辑、信息层级、数据图表和可读性。
不要为了视觉效果牺牲文字清晰度。
每页标题必须可单独串成故事线。
代码结构请按页拆分函数，便于后续针对单页修改。
如果输入是内容型 PPT MD，请先输出视觉导演优化版 MD，再进入 HTML 生成。
生成前请同时读取页面结构导演稿、当前设计语言的 design.md、style-card、slide-type-map、contact sheet，以及 final-layout-review.md。
页面结构导演稿不是视觉风格来源，不要要求它包含颜色、字体、母版元素或截图页型。
页面结构导演稿中的“左侧/右侧/顶部/底部/中部/侧边”等是版式说明，不是上屏文案；生成 HTML/PPTX 前必须清洗这些区域前缀。
contact sheet 只用于风格靠齐，不要求逐页复刻；但颜色、字体、母版固定元素、组件语法和信息密度必须与 design.md 一致。
请根据 design.md 在代码中实现对应的母版函数、模块函数和连接线/箭头函数。
请在 HTML/PPTX 生成前先产出生成就绪导演稿，补齐输出格式、画布参数、坐标/比例、样式 token、组件映射、连接规则和生成优先级。
请先生成 HTML 预览和 contact sheet，并按产品解决方案专家 + PPT 专家双视角做最终布局审查。
HTML 审查通过后再映射 PPTX；PPTX 输出后必须导出实际渲染截图，与 HTML/PNG 预览对比后再次审查。低于 80 分或存在 P0 问题时，不得作为最终成果交付。
```
