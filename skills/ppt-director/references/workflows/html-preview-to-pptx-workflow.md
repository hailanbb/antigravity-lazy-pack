# HTML Preview To PPTX Workflow

本工作流用于把 PPT 生成拆成两个阶段：

1. 先按页面结构稿和设计语言生成 HTML 设计预览。
2. 人工确认后，再把 HTML 设计映射成可编辑 PPTX。

它解决的问题是：直接从 MD 生成 PPTX 时，很难快速判断“像不像、挤不挤、主视觉够不够强”；HTML/CSS 更适合先做视觉排版预演。

## 1. 使用时机

当用户要求生成、重做、压测或优化 PPTX，并且需要遵循某个设计语言时，默认启用本工作流。

适用场景：

- 政务汇报、解决方案、产品方案、路演、培训课件。
- 用户提供了 `design.md`、PPT 模板、截图样例或指定风格。
- 用户对“设计语言一致性”和“页面效果”敏感。
- 页面结构复杂，需要先看版式再落 PPTX。

## 2. 阶段一：HTML 设计预览

输入：

- 经过页面视觉导演优化后的标准交付文档或逐页页面结构导演稿。
- 明确的 `页面描述_优化版.md`。如果只有用户原始 MD 或内容型 PPT MD，必须先返回 `visual-director-optimization.md`，不得直接生成 HTML。
- 设计语言对表。
- 生成就绪导演稿。它必须已经读取当前设计语言，并补齐输出格式、画布参数、区域坐标/比例、样式 token、字体字号参数、组件映射、连接规则和生成优先级。
- `design.md` / `style-card.md` / `slide-type-map.md` / `contact-sheet.png`。
- 当前设计语言在 `registry.yml` 中声明的固定资产，例如 digital-zhejiang 的 `assets.city_skyline`。
- 图片、截图、图标或占位符路径。

输出：

- `index.html` 或一组 HTML slide 页面。
- CSS 变量和组件样式。
- 每页 16:9 预览图。
- contact sheet。
- 设计检查说明。
- 最终布局审查报告。

HTML 设计要求：

- 每页画布固定为 16:9，例如 `1280x720`。
- CSS 必须从当前设计语言抽象变量：颜色、字体、字号、字重、标题区、母版元素、组件语法、页型节奏。
- 固定母版资产必须使用原始文件路径，例如 `city-skyline.png`；不得用 CSS/SVG 或 AI 重新生成。
- 页面内容严格来自页面结构导演稿，不擅自改标题、模块顺序和表达重点。
- 如果输入仍是内容型 PPT MD，必须先执行 `visual-director-optimization.md`，再生成 HTML。
- 页面结构导演稿中的区域词只用于排版定位；`左侧：`、`右侧：`、`顶部：`、`底部：`、`中部：`、`侧边：` 等前缀不得作为 HTML 可见标题。
- 优先执行生成就绪导演稿中的输出格式、画布参数、坐标/比例、组件映射和连接规则；若执行时需要调整，必须在审查报告中说明。
- 主视觉优先，文字辅助。
- 必须落实页面结构稿中的模块间关系：并列、支撑/底座、总分、贯穿型能力、侧挂治理层等。
- 必须落实面积预算：主视觉占主体区 >= 55%；流程路径页主视觉 >= 60%；辅助模块 <= 30%。
- 必须落实字体字号参数：字体、字号、字重来自 `design.md`，不得通过低于档位的字号强行塞内容。
- 必须落实固定资产引用：digital-zhejiang 正文页右下城市剪影直接引用 `references/styles/digital-zhejiang/assets/city-skyline.png`，保持右下锚定和透明背景。
- 贯穿型能力不得与流程并列抢主视觉，应以底座、背景条、侧挂治理层或轻量护栏呈现。
- 每页缩小后仍能看出结构。
- 不要求每页逐张复刻模板截图，但整体必须像同一套设计系统新增页面。

## 3. HTML 最终布局审查门禁

HTML 预览生成后，必须读取并执行：

`references/workflows/final-layout-review.md`

审查角色：

- **产品解决方案专家**：审查页面是否说清业务价值、方案边界、场景闭环、可信依据和管理抓手。
- **PPT 专家**：审查页面是否有视觉重心、主体区域是否铺开、信息密度是否合适、主视觉是否承担表达逻辑、缩略图是否看得出结构。
- **微观布局审查**：审查区域内元素是否均匀铺设，图文高度是否一致，同组元素间距、卡片内边距、文字基线、连接线、页脚安全区和品牌角标避让是否合格。
- **模块关系审查**：审查多个模块之间是否有明确视觉关系，是否出现无关系堆叠、双主视觉冲突、贯穿型能力位置错误。
- **主辅面积审查**：审查主视觉和辅助内容的面积比例是否符合页面类型，尤其是流程路径页。
- **设计语言审查**：审查是否匹配当前 `design.md`、style-card、slide-type-map 和 contact sheet 气质。
- **字体字号审查**：审查字体、字号、字重是否符合当前设计系统，尤其检查 digital-zhejiang 的微软雅黑和大标题/小标题/内容文字档位。
- **固定资产审查**：审查右下城市剪影等母版资产是否引用原图，是否被重绘、替换或遮挡。

HTML 审查输出必须包含：

- 总分，满分 100。
- 产品解决方案专家意见。
- PPT 专家意见。
- 设计语言一致性意见。
- 模块关系审查。
- 主视觉面积审查。
- 页面类型专项审查。
- P0/P1/P2 修改清单。
- 是否通过：`通过 / 不通过`。

通过标准：

- 总分必须 >= 80。
- 不存在任何 P0 问题。
- 每页缩略图状态下能看出标题区、主体区、底部区和主视觉方向。
- 页面可见文案中不得出现由结构稿带入的 `左侧：`、`右侧：`、`顶部：`、`底部：`、`中部：`、`侧边：` 等区域前缀。
- 不存在双主视觉冲突。
- 不存在模块无关系堆叠。
- 流程路径页中，流程主视觉必须占主体区 60% 以上；价值点只作为底部收束或侧边点缀。
- 字体字号必须符合设计系统；不得使用微软雅黑 Light、宋体或特殊字体；正文不得低于设计系统内容文字档位。
- 固定资产必须引用 registry 中的原图；digital-zhejiang 城市剪影不得自行生成或重绘。

未通过时：

- 不得进入 PPTX 映射。
- 只能输出修改清单和下一轮 HTML 修正计划。
- 必须先修 HTML，再重新截图和审查。

## 4. 人工确认门禁

HTML 预览完成且最终布局审查通过后，必须先展示：

- HTML 文件路径。
- contact sheet 路径或关键截图。
- 设计语言检查摘要。
- 最终布局审查摘要。
- 需要用户确认的问题，例如“是否按这版进入 PPTX 映射？”

在用户明确确认前，不执行 PPTX 生成。HTML 审查未通过时，即使用户未明确要求强行继续，也不要进入 PPTX。

可以继续迭代 HTML：

- 调整某一页版式。
- 降低信息密度。
- 强化主视觉。
- 更换页型。
- 替换截图或图片。
- 调整页面节奏。

## 5. 阶段二：PPTX 映射

用户确认 HTML 设计后，再执行 PPTX 映射。

原则：

- 不要默认把整页 HTML 截图塞进 PPT。
- 标题、正文、模块文字、基础形状、箭头、矩阵、分层架构、流程节点应尽量映射为 PPT 原生可编辑元素。
- 复杂背景、复杂插画、难以原生复刻的局部视觉，可以局部栅格化为图片。
- 如果用户明确要极高视觉还原且不关心可编辑性，才允许整页截图式 PPT，并说明取舍。

PPTX 映射对象：

- HTML 容器 -> PPT 形状或组合。
- HTML 文本 -> PPT 文本框。
- CSS 矩形/标签/模块 -> PPT 矩形。
- SVG/图标 -> PPT 图片或矢量。
- CSS 网格/矩阵 -> PPT 表格或形状网格。
- 流程线/箭头 -> PPT 原生线条或箭头。
- 截图/产品图 -> PPT 图片框。

## 6. PPTX 校验与最终审查

输出 PPTX 后必须检查：

- 页数是否正确。
- PPTX 是否可打开。
- 是否保留可编辑文本和形状。
- 关键页面是否与 HTML 预览一致。
- 是否仍符合设计语言。
- 是否存在明显遮挡、越界、文字过密。
- 字体、字号、字重是否仍符合设计系统，是否发生字体替换、换行、溢出或遮挡。
- 城市剪影等固定资产是否仍是原图、位置正确、未被重绘或替换。
- 是否保留 HTML 阶段确认的模块关系和主辅面积比例。
- 是否在 PPTX 映射时引入新的双主视觉冲突、箭头错位或区域词误上屏。
- 是否已导出 PPTX 的实际渲染截图，并与 HTML/PNG 预览进行视觉一致性对比。

随后必须再次执行最终布局审查：

- 用 PPTX 导出的截图或实际预览图，而不是仅看代码。
- 将 PPTX 渲染截图与 HTML/PNG 预览放在同一张对比图或 contact sheet 中，检查字体换行、模块位置、箭头方向、连接关系、母版元素和主辅面积是否漂移。
- 同时给出产品解决方案专家意见和 PPT 专家意见。
- 总分 >= 80 且无 P0 问题，才允许输出为最终 PPTX。
- 若不通过，必须回到 HTML 或 PPTX 映射代码修改，不得宣称完成。

### 6.1 PPTX 渲染一致性门禁

PPTX 的最终审查必须看“打开后的样子”，不能只看文件结构或代码对象。推荐流程：

1. 从 PPTX 导出实际渲染截图。可用 PowerPoint 导出、macOS QuickLook `qlmanage -t`、LibreOffice `--convert-to png`、或其他能代表用户打开效果的工具。
2. 将 PPTX 渲染截图与已确认的 HTML/PNG 预览并排生成对比图。
3. 检查以下项目：
   - 标题、判断句、模块文字是否发生换行、溢出或字体替换，字号是否仍在设计系统档位内。
   - 流程线、箭头和连接点是否仍指向正确对象。
   - 主视觉是否仍占主体区主导位置，辅助模块是否没有抢视觉。
   - 母版元素、页脚、安全区、角标、背景图形是否位置正确。
   - 固定资产是否仍引用原图，例如 `city-skyline.png`，没有被替换成生成图。
   - 区域词是否没有误上屏。
4. 如果差异明显，判定 PPTX 不通过。优先修复原生映射；如果短期无法修复且用户更重视视觉还原，可以输出：
   - `视觉一致版`：整页截图或局部栅格化，保证和 HTML/PNG 一致。
   - `可编辑原生版`：文本和形状可编辑，但需标明仍需人工校准。

交付时必须说明采用哪一种作为最终版。默认不得把视觉明显错误的可编辑原生 PPTX 当作最终文件。

## 7. 禁忌

- 不要跳过 HTML 预览直接生成最终 PPTX，除非用户明确要求快速草稿。
- 不要跳过 HTML/PPTX 最终布局审查直接交付。
- 不要只做设计语言检查，不做产品解决方案和 PPT 版式检查。
- 不要在 HTML 阶段改写内容逻辑。
- 不要把布局区域说明当成上屏文案。
- 不要把设计语言写回页面结构导演稿。
- 不要跳过页面描述优化，直接把用户原始 MD 生成 HTML/PPTX。
- 不要让流程、价值点和说明模块平均分配面积。
- 不要把贯穿型能力塞在流程之间。
- 不要把整页 HTML 截图作为默认 PPTX 输出。
- 不要为了还原视觉牺牲标题、正文和模块文字的可编辑性。
- 不要只验证 PPTX 文件可打开；必须验证 PPTX 实际渲染图。
- 不要在未做 PPTX 与 HTML/PNG 渲染对比时宣布 PPTX 审查通过。
- 不要把可编辑但视觉漂移明显的 PPTX 称为最终成稿。

## 8. 对用户的默认说明

当用户要求生成 PPTX 时，应简短说明：

> 我会先生成 HTML 设计预览和 contact sheet，并用产品解决方案专家 + PPT 专家双视角做最终布局审查。HTML 审查通过且你确认后，再映射为可编辑 PPTX；PPTX 也会再次审查，通过后才交付。
