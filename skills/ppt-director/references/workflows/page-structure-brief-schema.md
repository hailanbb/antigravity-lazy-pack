# Page Structure Brief Schema

页面结构导演稿用于把“逐页内容”整理成可生成的 PPT 单页描述。它只描述页面判断、信息组织、主视觉证明对象、上屏文案、管理价值和质量约束，不绑定任何具体设计语言。

生成 PPTX、Canva、HTML/Marp 或 AI 生图 Prompt 前，每一页都必须先具备此结构。具体颜色、字体、母版、组件、页型和截图靠齐规则，必须在生成阶段读取所选 `design.md` 后再决定，并合成为 `生成就绪导演稿`。

## 核心原则

一页好的 PPT 不是“标题 + 大段文字”，而是：

> 一个清晰结论 + 一个主视觉证明对象 + 少量解释文案 + 明确版式约束。

给模型描述单页时，必须同时回答四件事：

1. 这一页要表达什么判断。
2. 这一页采用什么版式。
3. 这一页的主视觉应该是什么。
4. 哪些内容必须出现，哪些表达方式不要出现。

禁止只写“高级、简洁、科技感、正式”等抽象风格词。风格词只能作为约束补充，不能替代页面结构。

## 页面类型

每一页必须选择一个主页面类型：

| 页面类型 | 适合表达的问题 | 推荐主视觉 |
| --- | --- | --- |
| 分层架构页 | 平台由哪些层组成 | 上下分层架构图 |
| 业务流程页 | 一件事如何完成 | 横向流程、闭环流程、连续业务链路 |
| 能力体系页 | 平台有哪些能力 | 中心平台 + 周边能力模块 |
| 对比分析页 | 几种模式有什么差异 | 矩阵表、左右对比、三栏对比 |
| 场景价值页 | 能解决什么业务问题 | 场景链路 + 价值卡片 |
| 实施路径页 | 分几步建设 | 时间轴、阶段路径图 |
| 案例成效页 | 做出了什么结果 | 案例卡 + 指标/成效 |
| 机制说明页 | 规则如何运转 | 机制环路、角色关系图 |

如果一页兼具多个类型，选择主视觉承担主要解释责任的类型。例如“业务流程 + 场景价值”应以流程页为主，价值放在底部或侧边。

## 标准格式

```markdown
# 页面标题
[页面标题，最好是“主题 + 判断/动作”。不要只写名词。]

# 本页核心结论
[一句话说明本页要让观众记住什么，不超过 40 字。不是标题重复，也不是背景说明。]

# 页面类型
[分层架构页 / 业务流程页 / 能力体系页 / 对比分析页 / 场景价值页 / 实施路径页 / 案例成效页 / 机制说明页]

# 页面版式
- 16:9 PPT 单页
- 顶部：[标题 + 核心判断 / 章节编号 / 其他顶部信息]
- 主体：[主布局，例如左侧流程 + 右侧说明、上下分层架构、四象限矩阵、中心平台 + 周边能力]
- 侧边：[如有侧挂模块、案例截图、风险提示、监测模块，写清位置；没有则写“无”]
- 底部：[总结语 / 价值标签 / 数据来源 / 页码]
- 信息密度：[低 / 中 / 高；说明文字是否需要克制]

# 主视觉设计
[说明这一页主要看什么图。主视觉必须承担解释逻辑的作用，而不是装饰。]

# 主视觉内容
## 区域/层级/步骤 1
- [必须出现的元素或短词]
- [必要解释]

## 区域/层级/步骤 2
- [必须出现的元素或短词]
- [必要解释]

## 区域/层级/步骤 3
- [必须出现的元素或短词]
- [必要解释]

# 模块间关系
- 主模块：[页面最重要的主视觉对象]
- 辅助模块：[侧边说明 / 价值点 / 备注 / 监测模块等]
- 关系类型：[并列 / 支撑底座 / 总分 / 贯穿型能力 / 侧挂治理层 / 其他]
- 表达方式：[等宽等高 / 底部横跨 / 主模块面积 >= 从属模块 2 倍 / 背景条 / 护栏 / 连接线]
- 禁止风险：[不要多个模块上下堆叠但无关系暗示；不要形成双主视觉]

# 主视觉与辅助内容面积预算
- 主视觉占主体区：[建议 >= 55%；流程路径页建议 >= 60%]
- 辅助模块占比：[建议 <= 30%]
- 价值总结占比：[只做底部收束或侧边点缀]
- 是否存在双主视觉冲突：[是/否；如是，说明拆页或降级策略]

# 页面文案
## 模块 1
[上屏短文案。模块标题尽量 4-8 字，说明尽量 12-24 字。]

## 模块 2
[上屏短文案。]

## 模块 3
[上屏短文案。]

# 页面想表达的管理价值
- [价值 1]
- [价值 2]
- [价值 3]

# 页面底部总结语
价值：[一句话收束本页观点。]

# 生成风险提示
- [是否存在模块关系不清]
- [是否存在主视觉占比不足]
- [是否存在价值点抢主视觉]
- [是否存在贯穿型能力位置错误]
- [是否存在连接线/箭头方向风险]

# 出图要求
- 层级关系一眼可读
- 缩小后仍能看出页面结构
- 主视觉优先，文字辅助
- 不要文字过密
- 不要生成普通模板页
- 不要复杂系统拓扑图，除非明确要求
- 中文清晰，适合正式汇报
```

## 写作规则

- 页面结构导演稿只管“内容结构”，不写具体设计语言名称。
- 不写具体颜色、字体、母版元素、品牌角标、城市剪影、截图样例页型。
- `顶部`、`主体`、`侧边`、`底部`、`左侧`、`右侧`、`中部`、`上方`、`下方` 等词只说明排版区域，不是上屏文案。
- 如果为了说明区域写成 `左侧：公共数据共享平台`，生成阶段必须把可见标题清洗为 `公共数据共享平台`；不要让 `左侧：` 出现在 HTML 或 PPTX 页面里。
- `页面文案` 栏才是上屏短文案；`页面版式` 和 `主视觉内容` 中的区域说明用于排版和构图。
- 标题要带判断或动作，例如“AI+ 体系架构：以 Agent Runtime 为核心”，不要只写“AI+ 体系架构”。
- 核心结论是方向盘，必须清楚说明这一页要证明什么。
- 页面类型决定主视觉形式，不能缺失。
- 主视觉必须是页面的视觉中心，不能只是几个卡片平铺。
- 页面文案是“上屏文案”，不是背景解释；长解释放到素材说明或演讲备注。
- 模块标题尽量 4-8 字，模块说明尽量 12-24 字。
- 每页只能服务一个核心结论；若出现两个核心结论，拆页。
- 当页面存在多个模块时，必须通过“模块间关系”说明它们的视觉关系。
- 并列模块应等宽等高；支撑/底座模块应置于底部并横跨；总分关系中主模块面积应至少为从属模块 2 倍。
- 每页应只有 1 个明确主视觉。主视觉占主体区域建议 >= 55%；流程路径页建议 >= 60%；辅助模块建议 <= 30%。
- 如果流程、价值点、侧边说明各占类似面积，应判定为双主视觉冲突并重写版式。
- 监测中心、审核中心、安全中心、运营中心等贯穿型能力，应作为底座、背景条、侧挂治理层或轻量护栏，不应塞在流程中间。
- 出图要求必须写“不需要什么”，例如不要文字墙、不要普通模板页、不要复杂拓扑图。

## 与生成就绪导演稿的关系

页面结构导演稿不是最终代码生成 Prompt。它负责“内容结构正确”，但不负责“执行参数完整”。

进入 HTML/SVG/python-pptx/PPTX 生成前，必须另行生成 `生成就绪导演稿`，补齐：

- 输出格式：HTML+CSS、SVG、python-pptx、HTML+PPTX。
- 画布参数：1280×720 或对应 16:9 尺寸、安全区、标题区、主体区、底部区。
- 设计系统参数：从当前 `design.md` 读取的色彩、字体、母版元素、组件形态、线条和箭头样式。
- 区域坐标或比例：主视觉、辅助模块、底部价值区的 x/y/w/h 或百分比。
- 组件映射：内容节点如何落到设计系统组件。
- 连接规则：箭头起点、终点、方向和禁止连接。
- 生成优先级：例如主视觉面积和连接关系正确优先于局部装饰美观。

这意味着：

- 页面结构导演稿保持可换风格。
- 生成就绪导演稿考虑当前设计系统，直接服务生成。
- 如果评审指出“描述很好但缺格式/坐标/样式参数”，应补 `生成就绪导演稿`，不要把风格参数硬塞回页面结构导演稿。

## 类型模板

### 分层架构页

```markdown
# 页面标题
XXX 体系架构：以 XXX 为核心，XXX / XXX 双支撑

# 本页核心结论
通过分层架构和平台支撑，实现 XXX 能力的统一建设、治理和复用。

# 页面类型
分层架构页。

# 页面版式
- 16:9 PPT 单页
- 顶部：标题 + 核心结论
- 中部：上下分层架构图
- 核心层居中高亮
- 支撑平台放在核心层下方
- 左右两侧放横向保障能力
- 底部放一句价值总结

# 主视觉设计
用“分层架构 + 核心运行层 + 平台支撑 + 左右护栏”的方式呈现。不要画成复杂网络拓扑图，不要把所有能力平均铺开。

# 主视觉内容
## 1. 场景层
- 场景 1
- 场景 2

## 2. 核心运行层
- 能力 1
- 能力 2

## 3. 平台支撑层
- 平台 A
- 平台 B

# 页面底部总结语
价值：以核心层为枢纽，将业务、模型、数据和基础设施组织成可治理、可复用的平台体系。
```

### 业务流程页

```markdown
# 页面标题
XXX 业务流程：从 XXX 到 XXX 的闭环办理链路

# 本页核心结论
将分散操作串联成一条可跟踪、可协同、可闭环的业务链路。

# 页面类型
业务流程页。

# 页面版式
- 16:9 PPT 单页
- 顶部：标题 + 核心判断
- 中部采用横向流程或连续业务链路
- 每一步下方放一句简短说明
- 底部放闭环价值总结

# 主视觉设计
用连续业务链路展示“发起 -> 匹配 -> 校验 -> 办理 -> 反馈”的全过程。如果强调体验，使用连续插画；如果强调管理规则，使用流程图。
```

### 能力体系页

```markdown
# 页面标题
XXX 能力体系：围绕 XXX 构建一体化支撑能力

# 本页核心结论
以统一平台为中心，沉淀可复用能力，支撑多场景快速落地。

# 页面类型
能力体系页。

# 页面版式
- 16:9 PPT 单页
- 中心放核心平台或核心能力
- 周边分组放能力模块
- 右侧或底部放管理价值
- 不要做成普通卡片堆叠

# 主视觉设计
用“中心平台 + 周边能力环绕”的结构展示。中心对象要突出，周边能力按类别分组。
```

### 对比分析页

```markdown
# 页面标题
XXX 模式对比：从管理效率、市场活力和安全可控三个维度分析

# 本页核心结论
不同模式各有适用边界，应根据治理目标、资源基础和市场成熟度选择。

# 页面类型
对比分析页。

# 页面版式
- 16:9 PPT 单页
- 顶部：标题 + 核心结论
- 中部使用对比矩阵
- 横向为不同模式
- 纵向为比较维度
- 底部放选择建议
```

### 实施路径页

```markdown
# 页面标题
XXX 实施路径：分阶段推进平台建设与场景落地

# 本页核心结论
按照“夯基础、建能力、推场景、强运营”的节奏逐步落地。

# 页面类型
实施路径页。

# 页面版式
- 16:9 PPT 单页
- 中部采用横向时间轴
- 每阶段一个节点
- 节点上方写阶段名称
- 节点下方写关键任务和交付物
- 底部放总体推进逻辑
```

## 快速检查清单

- 是否只有一个核心结论？
- 是否说明页面类型？
- 是否说明主视觉是什么？
- 是否说明页面版式？
- 是否区分“上屏文案”和“解释性背景”？
- 是否把区域词作为排版指令，而不是可见标题？
- 是否写了管理价值和底部总结语？
- 页面缩小后是否仍能看出结构？
- 是否避免大段文字、普通卡片堆叠和复杂拓扑图？

## 一句话方法

> 标题定主题，结论定判断，版式定骨架，主视觉定表达，模块定内容，价值定收束，约束定质量。
